# ADR 012: Manual Review Inbox (Domain Layer)

## Context
Transactions generated by the LLM Fallback (where `isConfident = false`) or transactions missing a known category must not silently pollute the user's ledger. They need to be routed to a "Review Inbox" for manual user validation. Since UI implementation is deferred to Phase C, we need to ensure the domain and data layers are fully capable of serving this inbox state reactively.

## Decision
1. **State Exposure**: Added `observeUnreviewedTransactions()` to `TransactionRepository`. This exposes a Kotlin `Flow` directly from the SQLDelight `selectUnreviewedTransactions` query.
2. **Reactivity**: By using `sqldelight.coroutines.asFlow()`, any new transaction parsed and inserted by the Orchestrator with `is_reviewed = 0` will instantly emit a new list to observers.
3. **Action Support**: The `reviewTransaction(id, categoryId)` function (built in Phase A5) provides the action layer to clear items from this inbox and apply the final categorization.

## Consequences
- The UI layer (when built) will simply collect this `Flow` and map it to UI states. No manual polling required.
- The boundary between `core-ledger` (Data) and `feature-inbox` (UI) remains strictly decoupled.
