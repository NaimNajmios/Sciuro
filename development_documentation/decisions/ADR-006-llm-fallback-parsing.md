# ADR 006: LLM Fallback Parsing

## Context
Our deterministic parsers (regex) are anti-fragile but inherently rigid. When a bank drastically changes its notification format, or a new unsupported e-wallet is used, the regex will fail to extract confidence data (like Merchant). We need a fallback mechanism that is intelligent enough to parse natural language notifications into our `StructuredDraft` schema without developer intervention.

## Decision
1. **Groq Llama 3 8B**: We chose Groq's cloud API for the fallback layer because of its unmatched inference speed (often <500ms) and generous rate limits for small models. The Llama 3 8B model is more than capable of extracting JSON from a short notification string.
2. **Opt-In Architecture**: To preserve privacy, the `LlmFallbackParser` skips execution entirely if an API key is not provided. Users who do not opt-in will simply have their unrecognized notifications routed to the Manual Review Inbox.
3. **Pipeline Fallback**: We introduced `SciuroParserPipeline`. It attempts the `DeterministicParser` first. If it returns a confident draft, the pipeline terminates. If it fails or returns `isConfident = false`, it invokes the `LlmFallbackParser`.
4. **Mandatory Review**: Any draft generated by the LLM is explicitly marked with `isConfident = false`. This guarantees that an LLM-parsed transaction will never be booked silently to the ledger; it will always enter the Manual Review Inbox for human verification.

## Consequences
- Fast, highly accurate extraction for new formats.
- Complete privacy preservation via the opt-in requirement.
- A human-in-the-loop safety net preventing AI hallucinations from corrupting the ledger.
